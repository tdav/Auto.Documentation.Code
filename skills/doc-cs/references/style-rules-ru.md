# Стиль-правила для summary/remarks/exception

Чек-лист «как НЕ писать» и фразы-замены. Цель — чтобы `<summary>` не вырождался
в пересказ имени метода.

## Стопфразы (признак мусорного summary)

Если черновик `<summary>` начинается с одной из этих фраз и дальше идёт лишь
перефразирование имени метода — переписать:

- «Получает …» (для `GetXxx`)
- «Устанавливает …» (для `SetXxx`)
- «Возвращает …» (когда дальше только тип)
- «Метод для …», «Функция для …»
- «Создаёт объект …» (для конструктора без контекста)
- «Вставляет запись …» (для `Insert` без указания таблицы и смысла)
- «Обновляет данные …», «Удаляет элемент …»

## Фразы-замены

Вместо тавтологии — что делает **и зачем**, с привязкой к домену:

| Плохо | Хорошо |
|---|---|
| «Получает пользователя по id» | «Загружает запись `DICT.SV_USER` по `SV_ID`; null, если запись помечена удалённой (`SV_RECORDSTATUS = 0`)» |
| «Вставляет запись» | «Вставляет строку в `DICT.SV_ITEM` со всеми колонками, включая мультиязычные `SV_NAME00..05`» |
| «Обновляет запись» | «Обновляет `DICT.SV_ITEM` по `SV_ID`; для версионирования прежний `SV_ID` сохраняется в `SV_PREVIOUSID`» |
| «Возвращает список» | «Возвращает активные записи (`SV_RECORDSTATUS = 1`); пустой список, если справочник пуст» |

## Когда XML-doc оправдан, а когда — шум

- ✅ Публичный API (NuGet/SDK).
- ✅ Data-access слой, где маппинг колонок не очевиден.
- ✅ Нетривиальная бизнес-логика, инварианты, побочные эффекты.
- ❌ Тривиальные авто-свойства (`public int Id { get; set; }`) — `<summary>`
  здесь только шум.

## Чек-лист для DA-классов (обязательно в `<remarks>`)

При документировании метода доступа к данным укажи:

1. **Имя таблицы Oracle** (схема + таблица, напр. `DICT.SV_ITEM`).
2. **Тип операции** (`INSERT` / `UPDATE` / `SELECT` / `DELETE`).
3. **Затронутые колонки** (или «все N колонок» с перечислением ключевых).
4. **Bind-параметры** (`:p_ID`, `:p_GUID`, …).
5. **Условие выборки/обновления** (`WHERE SV_ID = :p_ID`).
6. **Семантику аудита и soft-delete**, если колонки присутствуют
   (`SV_USER`, `SV_DATEENTER`, `SV_RECORDSTATUS`).

## Общий тон

- Полные предложения, точка в конце.
- Без ALL-CAPS MUST/NEVER в тексте документации — объясняй причину.
- Русский язык; идентификаторы кода и имена колонок — как в коде.
