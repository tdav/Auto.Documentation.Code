# Дизайн skill `doc-folder`

> Spec для нового skill плагина `dotnet-doc-skills`.
> Дата: 2026-05-27. Статус: согласован, ожидает review перед планом реализации.

## 1. Назначение

`doc-folder` генерирует **один обзорный `README.md`** в корне указанной папки.
Файл служит навигационной «картой» мульти-проектной папки: его читает будущий
claude code (и человек), чтобы за один файл понять, какие проекты лежат в папке,
какие в них классы и что каждый класс делает.

Ключевое отличие от соседних skill: вывод — **обзорный, in-place, один файл**,
с описанием *логики и назначения* каждого класса, без полных сигнатур методов и
без таблиц параметров.

### Место в плагине и границы

| Skill | Что делает | Вывод |
|---|---|---|
| `doc-cs` | inline `///` XML-doc | правки в `.cs` |
| `doc-md` | детальная API-дока, один `.md` на класс | `documentation/api/**` |
| `doc-db` | реверс схемы Oracle из C# | `documentation/database.md` |
| **`doc-folder`** | **обзорная карта папки: проекты + классы с описанием логики** | **`<папка>/README.md`** |

**Когда отказаться (переключиться):**
- просят добавить `///` в `.cs` → `doc-cs`;
- просят детальную API-доку (по `.md` на класс, таблицы методов/параметров) → `doc-md`;
- просят описать схему БД → `doc-db`.

## 2. Параметры, триггеры, frontmatter SKILL.md

- **Обязательный аргумент** — путь к папке для документирования. Если аргумент
  не передан — спросить путь (в отличие от прочих, путь здесь по смыслу обязателен).
- Триггеры (`description`): «doc-folder», «сделай README для папки», «опиши
  проекты в каталоге», «карта кода папки», «обзор классов в папке» — даже без
  слова «skill». Явно указать, что это **не** `doc-cs`/`doc-md`/`doc-db`.
- `disable-model-invocation: true`.
- `allowed-tools: Read, Grep, Glob, Write, Bash, Task`.
- `argument-hint: путь к папке (мульти-проектной)`.
- Язык вывода — **русский**; имена типов и членов — как в коде.

## 3. Алгоритм (три прохода)

### Pass 1 — обнаружение (главная сессия, Glob + Grep)

1. Рекурсивно найти `.cs`: `Glob("<папка>/**/*.cs")`. **Исключить** `bin/`,
   `obj/`, сгенерированные файлы (`*.g.cs`, `*.Designer.cs`, `*.AssemblyInfo.cs`,
   `*.AssemblyAttributes.cs`).
2. Найти все `.csproj` (`Glob("<папка>/**/*.csproj")`). Для каждого проекта
   определить: имя (`.csproj` без расширения), корневую папку, root-namespace
   (по `<RootNamespace>` или имени проекта), `TargetFramework(s)`.
3. **Привязать каждый `.cs` к проекту** — по ближайшему `.csproj` вверх по дереву.
   `.cs` без проекта (если есть) собрать в псевдо-группу «вне проектов».
4. Для каждого `.cs` извлечь (Grep): `namespace`, public-типы
   (`class|interface|enum|record|struct`), базовый тип и интерфейсы из
   объявления, наличие существующего `/// <summary>`.
5. Построить реестр типов `ТипИмя → относительный путь` (для возможных
   внутренних якорных ссылок в README).

### Pass 2 — описание классов (LLM; опционально субагенты ≤ 10)

Для каждого public-типа прочитать тело и сформировать запись **~3-6 строк**:

- **Назначение** (1-2 предложения): что класс делает, его роль в системе/слое.
- **Логика** (1-2 предложения): как именно работает — ключевые операции,
  инварианты, побочные эффекты (например: версионирование через `SV_PREVIOUSID`,
  фильтр soft-delete, оркестрация вызовов).
- **Зависимости/использование** (1 строка): что инжектируется/используется и кем
  вызывается, если видно из кода.

Запрещено: таблицы параметров, перечисление сигнатур методов, выдуманные факты.
Если из кода и `///` ничего не выводится — короткое фактическое описание по имени
и типу, без домыслов. Стиль фиксируется в `references/class-summary-style-ru.md`,
который **каждый субагент обязан прочитать** перед генерацией.

### Pass 3 — сборка единого README (главная сессия)

1. Собрать `README.md` по шаблону `assets/readme.md.template` (структура — §4).
2. Если существуют `documentation/api/` (от `doc-md`) или `documentation/database.md`
   (от `doc-db`) — добавить на класс/проект ссылки на соответствующие `.md`
   (мост к детальной доке). Если их нет — секцию ссылок не добавлять.
3. Записать через `Write` в `<указанная папка>/README.md`.

> **Параллелизм.** Pass 1 и Pass 3 — только в главной сессии (один выходной файл,
> конфликтов записи нет: блоки склеивает главная сессия). Pass 2 на больших папках
> распараллеливается субагентами `Task` волнами ≤ 10: каждому передаются его
> `.cs`-файлы и путь к `references/class-summary-style-ru.md`; субагент возвращает
> готовые markdown-блоки классов, главная сессия их компонует.

## 4. Структура README (вариант A — иерархия)

Выбран иерархический вариант: **Проекты → namespace → классы**. Внутри проекта
классы группируются по `namespace` (заголовок `### \`<namespace>\` (namespace)`).
Если в проекте один namespace — уровень группировки опускается, классы идут
списком сразу под проектом.

```markdown
---
title: <Имя папки> — обзор кода
type: folder-overview
generator: doc-folder
root: .
projects: [Sample.Dico, Sample.App]
generated: 2026-05-27
tags: [overview, csharp, navigation]
summary: Обзорная карта классов мульти-проектной папки.
---

# <Имя папки> — обзор кода

> Сгенерировано skill `doc-folder` (плагин `dotnet-doc-skills`).
> Назначение файла: карта папки для навигации (в т.ч. для AI-агента).
```

## 5. Формат записи класса

Заголовок `#### \`ИмяТипа\` — короткая роль`, далее строка-мета
(`Слой · namespace · ссылка на .cs`), далее абзац ~3-6 строк (§3, Pass 2).
Имена типов из этого же реестра можно оформлять якорными ссылками внутри README.

## 6. Перезапись

- `README.md` **всегда перезаписывается** при прогоне — это снимок текущего кода.
- Если `README.md` отсутствует — создать.
- Если `README.md` есть (в т.ч. написанный вручную) — заменить целиком.
- Поле `generator: doc-folder` во frontmatter сохраняется как метка происхождения
  (не управляет перезаписью, а лишь помечает источник файла).

## 7. Файлы skill и изменения манифестов

**Новые файлы:**
- `skills/doc-folder/SKILL.md`
- `skills/doc-folder/assets/readme.md.template`
- `skills/doc-folder/references/class-summary-style-ru.md`

**Изменения существующих:**
- `skills/doc-md/references/frontmatter-schema.md` — добавить секцию
  `type: folder-overview` (единый источник истины по frontmatter сохраняется).
- `.claude-plugin/plugin.json` — дополнить `description` (+`doc-folder`),
  поднять `version` 1.0.6 → 1.1.0.
- `.claude-plugin/marketplace.json` — дополнить описания плагина (+`doc-folder`).

## 8. Соглашения и запреты

- Вывод — русский; имена типов/членов/проектов — как в коде.
- Не выдумывать назначение, которого нет в коде/`///`/имени — лучше короче.
- Не воспроизводить сигнатуры методов и таблицы параметров (это `doc-md`).
- Не генерировать `///` в `.cs` (это `doc-cs`).
- Ссылки ставить только на существующие файлы/типы; относительные пути — от
  `README.md` в корне указанной папки.
- Исключать `bin/`, `obj/` и сгенерированные файлы из обхода.

## 9. Критерии приёмки

1. `doc-folder <папка>` создаёт один `README.md` в корне `<папка>`.
2. README содержит frontmatter с `type: folder-overview` и `generator: doc-folder`.
3. Перечислены все `.csproj`-проекты папки со сводной таблицей.
4. Каждый public-тип каждого проекта представлен записью с абзацем о
   назначении/логике (не одной строкой, не API-таблицей).
5. Повторный прогон полностью перегенерирует `README.md` (всегда перезаписывает).
6. Манифесты плагина обновлены, версия поднята до 1.1.0.
7. Skill согласован по стилю и frontmatter с `doc-cs`/`doc-md`/`doc-db`.
